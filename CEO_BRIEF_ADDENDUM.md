## PLAYBOOK 3: Bad Press / PR Crisis (Extended)

**Symptoms:**
- Negative article published
- Viral negative tweet
- User complaints escalating
- Competitor attack

### Response Strategy:
1. **DON'T respond immediately** (wait 2-4 hours)
2. Gather all facts
3. Consult founder on messaging
4. Craft measured response
5. Be factual, not defensive

### Response Guidelines:
- Acknowledge if criticism is valid
- Correct if information is wrong (with evidence)
- Explain context professionally
- Share what you're doing to improve
- Never attack the critic
- Take conversation offline if heated

### Response Template:
> Thanks for sharing your concerns about [issue].
> 
> [If valid]: You're right, and we're working on this. Here's our plan...
> 
> [If incorrect]: We appreciate the feedback, but want to clarify the facts: [factual correction with evidence]
> 
> [Always end with]: We're committed to [core value]. Here's how we're improving: [specific actions]
> 
> Happy to discuss further: [email/DM]

## PLAYBOOK 4: Development Blocker (Extended)

**Symptoms:**
- Critical feature not working
- Third-party API down
- Technical debt too high
- Can't meet deadline

### Immediate Actions:
1. Document the blocker clearly
2. Identify workarounds
3. Estimate delay impact
4. Notify founder if >2 day delay

### Problem-Solving Framework:
1. Define problem precisely
2. List all possible solutions
3. Evaluate each (pros/cons, time, cost)
4. Pick best option
5. Execute fast
6. Document decision

### Escalation Message:
> ⚠️ DEVELOPMENT BLOCKER - Day [X]
> 
> **Blocker:** [Specific technical issue]
> **Impact:** [Which feature/timeline affected]
> **Delay estimate:** [X days]
> 
> **Solutions considered:**
> 1. [Option A] - Pros: X, Cons: Y, Time: Z
> 2. [Option B] - Pros: X, Cons: Y, Time: Z
> 3. [Option C] - Pros: X, Cons: Y, Time: Z
> 
> **Recommendation:** [Option X] because [reasoning]
> 
> Need approval to proceed or discuss alternatives?

## PLAYBOOK 5: Budget Overrun (Extended)

**Symptoms:**
- Unexpected cloud costs
- API usage higher than planned
- Need to purchase additional services

### Immediate Actions:
1. Identify root cause of overrun
2. Calculate projected total overrun
3. Implement immediate cost controls
4. Notify founder if >$100 unexpected

### Cost Control Measures:
- Scale down non-essential services
- Optimize API calls
- Use caching aggressively
- Review for unnecessary resources

### Founder Notification:
> 📊 Budget Alert
> 
> **Issue:** [What caused overrun]
> **Current spend:** $X (budget: $Y)
> **Projected overrun:** $Z
> 
> **Actions taken:**
> 1. [Cost control 1]
> 2. [Cost control 2]
> 
> **Recommendation:** [Continue/scale back/approve additional spend]

## PLAYBOOK 6: Data Loss

**Symptoms:**
- Database corruption
- Accidental deletion
- Backup failure
- Storage failure

### Immediate Actions:
1. Assess scope of data loss
2. Stop any write operations
3. Check backup status
4. Notify founder immediately

### Recovery Steps:
1. Verify latest backup timestamp
2. Test backup restoration
3. Restore from last good backup
4. Verify data integrity
5. Document what was lost
6. Update users if needed

### Prevention:
- Regular backup testing
- Multiple backup locations
- Monitoring for storage limits
- Read-only replicas for critical data
